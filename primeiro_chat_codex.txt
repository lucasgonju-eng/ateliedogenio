Visão Geral

Sistema web “Ateliê do Gênio” com PHP 8.2 em camadas (Domain, Infrastructure, Http), Supabase (Postgres + PostgREST) e frontend estático com Tailwind via CDN.
Foco em controle de estoque, PDV, fluxo de caixa e dashboards separados para admin e vendedores.
Perfis: admin com acesso total; vendedor restrito a dados e vendas próprias via RLS no Supabase.
Objetivos e Escopo

Autenticação completa: login, logout, redefinição de senha com tokens e e-mail, gerenciamento de vendedores.
Gestão de estoque com SKUs, custos, preços, níveis de estoque e histórico de movimentações.
PDV com seleção de cliente (ou “cliente balcão”), registro da venda, cálculo de taxas e envio automático de recibos.
Fluxo de caixa monitorando entradas, taxas por método de pagamento e conciliações.
Dashboards separados: vendedor (kanban, metas, resumo) e admin (visão geral, lucro presumido, estoque crítico).
Arquitetura

Domain Layer: entidades, value objects, interfaces de repositórios/serviços e regras de negócio puras.
Infrastructure Layer: adaptadores para PostgREST, Supabase Auth, serviço de e-mail, cache e logging.
Http Layer: controladores REST, middlewares de autenticação/autorização, validações CSRF.
Comunicação com Supabase via HTTP client (ex.: Guzzle) encapsulado, assinando requisições com chaves seguras (service key para admin, session tokens para vendedor).
Configuração por variáveis de ambiente (.env) geridas por Dotenv; container interno de dependências para injeção em controladores e serviços.
Modelagem de Dados (Supabase)

roles(id, name, permissions, created_at); seed inicial: admin, vendedor.
users(id UUID, role_id, name, email, hashed_password, status, created_at, last_login_at).
customers(id UUID, name, email, phone, document, is_walk_in, created_at) com flag is_walk_in para “cliente balcão”.
products(id UUID, sku UNIQUE, name, description, cost_price, sale_price, stock_quantity, min_stock_alert, created_at, updated_at).
inventory_movements(id, product_id, user_id, type, quantity, unit_cost, reference, created_at) com tipos entrada, saida, ajuste.
payments_config(id, payment_method, fee_percentage, fee_fixed, allow_discount, max_discount_percentage, created_at).
sales(id, user_id, customer_id, status, payment_method, subtotal, discount_total, fee_total, total, estimated_profit, receipt_sent_at, created_at, updated_at).
sale_items(id, sale_id, product_id, quantity, unit_price, unit_cost, line_total, created_at).
cash_ledger(id, sale_id, payment_method, amount, fee_amount, net_amount, entry_type, created_at) para registrar entradas/ajustes.
Índices por user_id, status, payment_method para consultas rápidas; triggers atualizando stock_quantity em products.
RLS: políticas baseadas em auth.uid(); vendedores só leem/escrevem registros onde user_id = auth.uid(); admin identificado por claim customizada role = 'admin' com bypass total.
Regras de Negócio

Vendedor visualiza apenas vendas e movimentações próprias; acesso a estoque apenas leitura.
Admin cadastra/edita produtos, vendedores, configura taxas e descontos; pode visualizar todo o estoque e fluxo financeiro.
Métodos de pagamento suportados: crédito, débito, pix; taxas aplicadas conforme payments_config.
Lucro presumido = total − (soma unit_cost nos itens) − taxas do método de pagamento.
Descontos limitados pelo max_discount_percentage; validação no servidor e no frontend.
Ao finalizar venda: baixa de estoque, registro em inventory_movements, cálculo de taxas e lançamento em cash_ledger.
Recibos enviados via e-mail transactional (SMTP ou serviço) com resumo da compra.
Dashboards admin mostram KPIs de lucro, giro de estoque, alertas de estoque crítico; vendedor vê metas vs. vendas, pipeline de status.
Rotas e Funcionalidades (HTTP)

Autenticação: POST /auth/login, POST /auth/logout, POST /auth/password/forgot, POST /auth/password/reset.
Usuários: GET /users (admin), POST /users, PUT /users/{id}, PATCH /users/{id}/status.
Produtos: GET /products, POST /products, PUT /products/{id}, GET /products/{id}/movements.
Estoque: POST /inventory/adjust para ajustes administrativos, GET /inventory/alerts.
Vendas (PDV): POST /sales/draft, PUT /sales/{id}, POST /sales/{id}/finalize, GET /sales, GET /sales/{id}, POST /sales/{id}/receipt.
Fluxo de Caixa: GET /cash-ledger, POST /cash-ledger/adjustment.
Dashboards:
Admin: GET /dashboard/admin/summary, GET /dashboard/admin/profit, GET /dashboard/admin/stock-critical.
Vendedor: GET /dashboard/sales-board, GET /dashboard/personal-metrics.
Configuração de Pagamentos: GET/POST/PUT /payments/config (admin).
Fluxos Principais

Cadastro/Login: validação de credenciais contra Supabase Auth, criação de vendedores com link para reset de senha inicial.
Reset de senha: geração de token único com expiração, envio por e-mail, validação e atualização do hash.
PDV: vendedor cria venda em draft, seleciona cliente, adiciona itens (com consulta a estoque), aplica desconto dentro do limite, escolhe método de pagamento, sistema calcula taxas e lucro, finalização dispara baixa de estoque, registra ledger e envia recibo.
Fluxo admin: manutenção de catálogo, gerenciamento de usuários, configuração de taxas, análise de dashboards consolidados com filtros por período.
Integração com Supabase

Utilização do PostgREST com cabeçalhos apikey, Authorization: Bearer <token>; admin usa service key via backend seguro; vendedores usam JWT de sessão.
Middlewares PHP validam role e montam filtros (ex.: user_id=eq.<id>).
RLS complementar no banco para proteger acesso direto; claims customizadas configuradas no Supabase Auth.
Webhooks opcionais (Supabase Functions) para logging de segurança ou notificações.
Manter rotinas de refresh e revoke de tokens; armazenar sessões em cookies HttpOnly com Secure/SameSite.

Layout responsivo com Tailwind CDN, estrutura em HTML sem frameworks JS; interações leves com Alpine.js opcional se necessário.

Página de login com ilustrações inspiradas em Einstein, mantendo minimalismo e acessibilidade.
Regras de Lógica e Cálculos

Taxas: fee_total = subtotal * fee_percentage + fee_fixed; valores configuráveis por método.
Descontos: validação contra allow_discount e max_discount_percentage; logs de tentativas inválidas.
Estoque: decremento proporcional à quantidade vendida; movimentações guardam unit_cost para auditoria.
Lucro: estimated_profit = total - (∑unit_cost * qty) - fee_total; exibido em relatórios.
Recibo: template HTML minimalista, envio via serviço SMTP; registrar receipt_sent_at.
Fluxo de caixa: net_amount = total - fee_total, lançado em cash_ledger com entry_type='sale'.
Segurança

Hash de senha com password_hash (bcrypt ou Argon2id); enforce requisitos de senha.
Tokens e cookies HttpOnly, Secure, SameSite=Strict; regeneração de sessão pós-login.
CSRF tokens para formulários sensíveis; validação server-side.
Sanitização/validação de input com filtros estritos; prepared statements nas requisições auxiliares.
Rate limiting para login (ex.: bloquear após 5 tentativas por 15 minutos).
Logging de auditoria para operações críticas (criação de usuário, ajustes de estoque, alterações de taxas).
Monitoramento de integridade das políticas RLS e alertas de anomalias.

Projeto web “Ateliê do Gênio” baseado em PHP 8.2, Supabase (Postgres/PostgREST) e frontend HTML+Tailwind via CDN. Foco em arquitetura limpa, segurança, separação de responsabilidades e integração direta com Supabase para estoque, PDV, fluxo de caixa e dashboards diferenciados para admin e vendedores.

Arquitetura e Stack

Camadas PHP (Domain, Infrastructure, Http) isolando regras de negócio, integrações e controladores; dependências injetadas via container próprio; variáveis sensíveis carregadas do .env.
Supabase provê Auth, Postgres e PostgREST; políticas RLS configuradas por role (claim role=admin ou role=vendedor).
Infraestrutura inclui cliente HTTP (ex.: Guzzle) encapsulando chamadas ao PostgREST, serviço de e-mail, cache (Redis opcional) e logger.
Frontend puro com Tailwind CDN, componentes acessíveis, interatividade leve (JS vanilla ou Alpine opcional); design minimalista azul/amarelo com ícones científicos.
Modelagem de Dados (Supabase)

roles(id, name, permissions, created_at); seeds admin, vendedor.
users(id UUID, role_id, name, email UNIQUE, password_hash, status, created_at, last_login_at, metadata JSONB); relaciona-se a Supabase Auth via claim supabase.auth.users.
customers(id UUID, name, email, phone, document, is_walk_in BOOLEAN DEFAULT false, created_at); o “cliente balcão” tem is_walk_in=true.
products(id UUID, sku UNIQUE, name, description, supplier_cost NUMERIC, sale_price NUMERIC, stock INT, min_stock_alert, active BOOLEAN, created_at, updated_at); trigger atualiza stock conforme inventory_movements.
inventory_movements(id, product_id, user_id, type ENUM('entrada','saida','ajuste'), qty INT, unit_cost NUMERIC, reference TEXT, created_at).
payments_config(id, payment_method ENUM('credito','debito','pix'), fee_percentage NUMERIC, fee_fixed NUMERIC, allow_discount BOOLEAN, max_discount_percentage NUMERIC, created_at, updated_at).
sales(id, user_id, customer_id, status ENUM(...), payment_method ENUM, subtotal NUMERIC, discount_total NUMERIC, fee_total NUMERIC, total NUMERIC, profit_estimated NUMERIC, receipt_sent_at TIMESTAMPTZ, created_at, updated_at).
sale_items(id, sale_id, product_id, qty INT, unit_price NUMERIC, unit_cost NUMERIC, line_total NUMERIC, created_at); line_total=qty*unit_price.
cash_ledger(id, sale_id, user_id, payment_method, gross_amount, fee_amount, net_amount, entry_type ENUM('sale','adjustment'), created_at); user_id do autor do lançamento.
Índices: sales(user_id,status,created_at), inventory_movements(product_id,created_at), cash_ledger(payment_method,created_at).
RLS: vendedores acessam sales, sale_items, cash_ledger e inventory_movements filtrados por user_id; admin (claim role=admin) ignora RLS; polícias específicas para leitura e escrita.
Regras de Negócio

Vendedor vê apenas dados próprios; produtos e estoque apenas leitura.
Admin gerencia produtos, estoque, usuários e taxas/descontos; consegue realizar ajustes de caixa.
PDV aplica taxas conforme payments_config; valida desconto ≤ max_discount_percentage.
Lucro presumido = total - Σ(unit_cost*qty) - fee_total.
Ao finalizar venda: baixa estoque, cria inventory_movements (tipo saida), registra cash_ledger, atualiza sales.status, recalcula profit_estimated, dispara e-mail e atualiza dashboards.
Recibo enviado automaticamente ao cliente (ou vendedor recebe cópia para balcão).
Vendas em status sequencial: aberta → pagamento_pendente → paga → entregue; cancelada encerra sem baixa (se cancelada antes de paga, reversão de estoque).
Integração com Supabase

Comunicação apenas via PostgREST usando cabeçalho apikey e Authorization: Bearer; vendedor usa JWT de sessão; admin usa service key guardada no backend.
Claims customizadas: role e user_id inseridas via Supabase Auth hook.
Middlewares PHP traduzem operações de alto nível para requisições PostgREST (filtros ?user_id=eq.<uuid> etc.) e validam respostas.
Webhook opcional (Supabase Functions) para logging de eventos de segurança e notificações de estoque crítico.
Renovação de tokens, revogação em logout, armazenamento em cookies HttpOnly+Secure+SameSite=Strict.
Rotas Principais (além do OpenAPI fornecido)

Autenticação: POST /auth/login, POST /auth/logout, POST /auth/password/forgot, POST /auth/password/reset.
Usuários/Admin: GET /users (admin), POST /users, PUT /users/{id}, PATCH /users/{id}/status, GET /roles.
Clientes: GET /customers, POST /customers, PUT /customers/{id}, GET /customers/{id} (todas roles; vendedor limitado às próprias vendas em relatórios).

Detalhamento do OpenAPI

bearerAuth: JWT emitido pelo Supabase; renovado via refresh token no backend.
GET /products: admin; suporta filtros search, paginação limit/offset; cabeçalho X-Total-Count.
POST /products: admin; cria/atualiza produto; valida SKU único; responde 201.
GET /sales: vendedor; aplica RLS automática; aceita filtro status e paginação; retorna vendas em ordem decrescente de criação.
POST /sales: cria rascunho com itens; valida estoque mínimo; retorna sale_id e status='aberta'.
POST /sales/{id}/checkout: finaliza venda calculando fees, total, profit_estimated; em caso de conflito (estoque), retorna 409; desconto inválido gera 422.
GET /dashboard/vendor: retorna KPIs individuais (today_total, month_total, open_count, opcionais metas).
GET /dashboard/admin: KPIs consolidados, low_stock com lista de produtos abaixo de min_stock_alert.
Fluxos Operacionais

Registro e login de vendedores: admin cria usuário com role vendedor, dispara e-mail com link de definição de senha; vendedor autentica, backend salva sessão e tokens.
Reset de senha: POST /auth/password/forgot gera token e envia; POST /auth/password/reset valida token, atualiza hash.
PDV: vendedor cria rascunho, adiciona itens (verificando estoque), define cliente, prossegue para checkout; backend chama PostgREST para inventory_movements e sales com transação (via Supabase RPC ou API sequencial com tratamento de conflitos).
Fluxo de caixa admin: visualiza cash_ledger filtrado por período/método, exporta dados; pode registrar ajustes manuais (entry_type='adjustment').
Segurança

Hash de senha com Argon2id; requisitos mínimos de complexidade.
Rate limit login (ex.: 5 tentativas/15 min); logs de tentativas falhas.
CSRF tokens para POST/PUT/PATCH/DELETE; validação server-side.

Headers de segurança: Content-Security-Policy, X-Frame-Options, X-XSS-Protection, Strict-Transport-Security.
Auditoria: logs de criação/edição de produtos, ajustes de estoque, alterações de taxas, finalização/cancelamento de vendas.

Integração Supabase

Fluxo de checkout e ajustes deve usar funções Postgres expostas via /rpc porque envolve múltiplas escritas; uma chamada REST simples não garante atomicidade.
Criar funções SECURITY DEFINER com validações internas, mas sempre consultando auth.jwt() (auth.uid(), auth.role(), claims customizadas) para aplicar RLS manualmente e impedir bypass por usuários fora do backend.
Habilitar cabeçalho Prefer: tx=rollback em homolog e Prefer: tx=commit em produção para garantir transação única por chamada RPC; PHP deve definir Accept-Profile/Content-Profile corretos.
Checkout (fn_finalize_sale)

Assinatura sugerida: fn_finalize_sale(_sale_id uuid, _payment_method text, _discount_percent numeric default 0, _actor_id uuid default auth.uid()) returns json.
Etapas internas:
Selecionar venda FOR UPDATE garantindo status aberta/pagamento_pendente; validar user_id = _actor_id ou role = admin.
Validar método de pagamento e desconto contra payments_config; se ultrapassar limites, levantar RAISE EXCEPTION 'BUSINESS_RULE' com detail JSON (capturado como erro HTTP 422).
Carregar itens e produtos FOR UPDATE; abortar se products.stock < qty → RAISE EXCEPTION 'STOCK_CONFLICT' (HTTP 409).
Calcular subtotal, fee_total, total, profit_estimated e atualizar sales.
Inserir em inventory_movements (tipo saida), atualizar products.stock = stock - qty.
Inserir linha no cash_ledger com entry_type = 'sale'.
Registrar timestamp de envio previsto do recibo (retornar flag; envio real via PHP).
Retornar JSON com totais e indicadores (bate com schema CheckoutResponse).
PostgREST: POST /rpc/fn_finalize_sale com body {"_sale_id": "...", "_payment_method": "credito", "_discount_percent": 5}. PHP injeta _actor_id somente quando usa service key.
Tratamento de erros: mapear SQLSTATE personalizadas (P0001 com code e message em JSON) para HTTP 422/409/403.
Ajuste de Caixa (fn_register_cash_adjustment)

Parâmetros: _payment_method, _gross_amount, _fee_amount, _note, _actor_id.
Verificações: apenas admin (auth.role() = 'admin'), gross_amount ≠ 0, coerência com políticas financeiras.
Operações: inserir em cash_ledger com entry_type='adjustment', calcular net_amount = gross_amount - fee_amount, retornar registro.
PHP endpoint /cash-ledger/adjustment chama RPC e repassa resposta; erros de permissão retornam 403.

<?php

declare(strict_types=1);

namespace AtelieDoGenio\Http\Controller;

use AtelieDoGenio\Domain\Exception\BusinessRuleException;
use AtelieDoGenio\Domain\Service\AuthenticationService;
use AtelieDoGenio\Http\Response\JsonResponse;
use AtelieDoGenio\Infrastructure\Security\CsrfTokenManager;
use Psr\Http\Message\ResponseInterface;
use Psr\Http\Message\ServerRequestInterface;

final class AuthController extends BaseController
{
    public function __construct(
        private readonly AuthenticationService $authService,
        private readonly CsrfTokenManager $csrfTokens,
    ) {
    }

    public function login(ServerRequestInterface $request): ResponseInterface
    {
        $payload = $this->input($request);
        $email = trim((string) ($payload['email'] ?? ''));
        $password = (string) ($payload['password'] ?? '');

        if ($email === '' || $password === '') {
            return $this->error('VALIDATION_ERROR', 'Campos email e password sao obrigatorios.', 422);
        }

        try {
            $result = $this->authService->login($email, $password);
        } catch (BusinessRuleException $exception) {
            return $this->error($exception->errorCode(), $exception->getMessage(), 422);
        } catch (\Throwable $exception) {
            return $this->error('AUTH_ERROR', $exception->getMessage(), 500);
        }

        $claims = $result['claims'] ?? [];
        $userData = $result['user'] ?? [];

        $role = $result['role']
            ?? ($claims['user_metadata']['role'] ?? null)
            ?? ($claims['raw_user_meta_data']['role'] ?? null)
            ?? ($userData['user_metadata']['role'] ?? null)
            ?? ($userData['raw_user_meta_data']['role'] ?? null)
            ?? 'vendedor';

        $result['role'] = $role;
        $result['csrf_token'] = $this->csrfTokens->generate('api');

        return $this->json($result);
    }

    public function logout(ServerRequestInterface $request): ResponseInterface
    {
        $token = (string) $request->getAttribute('supabase_token', '');

        if ($token === '') {
            return $this->error('UNAUTHORIZED', 'Sessao nao encontrada.', 401);
        }

        $this->authService->logout($token);

        return JsonResponse::success(['message' => 'Sessao encerrada.'], 200);
    }

    public function forgotPassword(ServerRequestInterface $request): ResponseInterface
    {
        $payload = $this->input($request);
        $email = trim((string) ($payload['email'] ?? ''));

        if ($email === '') {
            return $this->error('VALIDATION_ERROR', 'Email eh obrigatorio.', 422);
        }

        try {
            $this->authService->requestPasswordReset($email);
        } catch (BusinessRuleException $exception) {
            return $this->error($exception->errorCode(), $exception->getMessage(), 422);
        }

        return $this->json(['message' => 'Se o usuario existir, um email foi enviado.'], 202);
    }

    public function resetPassword(ServerRequestInterface $request): ResponseInterface
    {
        $payload = $this->input($request);
        $token = (string) ($payload['token'] ?? '');
        $newPassword = (string) ($payload['new_password'] ?? '');

        if ($token === '' || $newPassword === '') {
            return $this->error('VALIDATION_ERROR', 'Token e nova senha sao obrigatorios.', 422);
        }

        try {
            $this->authService->resetPassword($token, $newPassword);
        } catch (BusinessRuleException $exception) {
            return $this->error($exception->errorCode(), $exception->getMessage(), 422);
        }

        return $this->json(['message' => 'Senha redefinida com sucesso.']);
    }
}
